# AI経費精算リーダー 開発プラン

領収書・レシートの画像 / PDF を **Claude API（AIの読み取り）** で読み取り、**勘定科目まで自動で仕分けして経費精算 CSV を出力**するWebアプリ。
個人事業主・フリーランス・中小企業の「レシートを1枚ずつ手入力して経費精算する」作業を消すことがゴール。`03_invoice_pdf_generator`（請求書を *作る* ツール）と対になるポートフォリオ第4弾。`02_scraping` と同じ **Streamlit** の作りで、ブラウザから使える。

---

## 1. ゴールと提供価値

- **課題**: 立替経費の精算で、レシートを1枚ずつ見て「日付・店名・金額・科目」を表計算や経費精算ソフトに手入力している。月末にまとめて発生し、軽減税率(8%)の判別や勘定科目の仕分けが地味に手間。
- **提供価値**: レシート画像をアップロードするだけで、日付・支払先・金額・税率別内訳・**登録番号(T番号)**を読み取り、さらに **勘定科目を自動で推定**。経費精算書 + 会計ソフト取込CSV をダウンロードできる。
- **差別化（勝ち筋）**:
  1. **勘定科目の自動仕分け** — ただのOCRでなく「旅費交通費 / 会議費 / 消耗品費 / 接待交際費」等を品目・店名から提案。経費精算で一番面倒な所を消す。
  2. **軽減税率(8%)混在レシート対応** — 飲食含むレシートの税率別集計。
  3. **ブラウザですぐ使える** — インストール不要。スマホのブラウザからレシートを撮ってそのままアップロードできる。

---

## 2. ターゲットと前提

- **ユーザー**: 個人事業主・フリーランス、中小企業で立替精算する従業員/経理、税理士事務所のスタッフ。
- **入力**: 領収書・レシート・店舗発行の請求書。スマホ撮影画像(JPG/PNG)、スキャンPDF、複数枚一括。フォーマットは店ごとにバラバラ（=ルールベースOCRが効かない領域 → AIが活きる）。
- **スコープ外（割り切り）**: 交通費のICカード履歴など「レシートが出ない経費」は対象外。まずは紙/PDF/画像のレシートに集中。
- **形態**: `02_scraping` と同じ **Streamlit のWebアプリ**。**ローカルで `streamlit run` して使う**（公開・デプロイはしない）。実力はREADMEのスクショ＋デモGIFで見せる。

---

## 3. 機能要件

### MVP（最初に作る）
1. アップロード: 画像/PDF を複数枚まとめてアップロード（`st.file_uploader`）。スマホのカメラからもOK。
2. AI読み取り: 1枚ずつ Claude API に渡して項目を抽出（後述スキーマ）。
3. **勘定科目の自動推定**: 店名・品目から経費区分を提案（確定は人）。
4. 結果テーブル: 抽出結果を表で表示。**セル直接編集 + 科目はドロップダウン**で人が直せる（`st.data_editor`）。
5. 要確認ハイライト: 金額が合わない・必須項目が空・科目の自信が低いものを目立たせる。
6. CSVダウンロード: 「経費精算書」用 / 「会計ソフト取込」用 を選んで出力（`st.download_button`）。

### 拡張（MVP後）
- **科目の学習/ルール化**: 「この店はいつも会議費」を記憶して次回から自動確定。
- 立替者・プロジェクト別の集計。
- 重複検出（同じレシートの二重計上を防止）。
- 月次の保管（原本+結果）。電子帳簿保存法の保存要件を意識。
- まとめ処理のコスト削減（Batches API で50%減、夜間一括）。

## 4. 技術構成（02と統一）

| レイヤ | 採用 |
|---|---|
| 言語 | Python 3.10+ |
| Web/UI | **Streamlit**（02と同じ。`streamlit_app.py` をルート配置） |

| PDF取り込み | pymupdf もしくは pdf2image（PDF→画像/base64） |

| デプロイ | クラウド（02と同じ運用に合わせる。※下記「デプロイ」参照） |

### なぜ Claude API（画像を直接読む）で一本化するか
- Claude（Sonnet 4.6 など）は **画像とPDFをそのまま入力**でき、レイアウトを理解した上で「項目の抽出」と「勘定科目の推定」を **1回の呼び出し**でできる。OCRで文字を取ってから整形して分類…という多段構成より工程がシンプルで、感熱紙の薄いレシートにも強い。
- **出力をJSONの形に固定**できる（Structured Outputs）。読み取り結果がいつも同じ形で返るので、そのまま表やCSVに流せる。
- **作りはシンプルに**: 他社AIに切り替えられる仕組みは作らず、Claude API 一本で素直に実装する。

## 5. AI読み取りの設計

### 入力
- 画像: そのまま base64 にして Claude へ渡す。
- PDF: 各ページを画像化して渡す（または `document` としてPDFを直接渡す）。
- 大きなスキャンはアップロード時に適切なサイズへ縮小（コスト＝画像トークン対策）。

### モデル選択
- **既定: `claude-sonnet-4-6`**（入力 $3 / 出力 $15 per MTok）。精度とコストのバランスが良く、レシートの量産処理に向く。
- **切替の選択肢**:
  - `claude-opus-4-8`（$5 / $25）— 崩れたレシートや科目推定の精度を最優先したい時に。
  - `claude-haiku-4-5`（$1 / $5）— 速度・単価最優先。単純なレシート向け。
- レシートは月数十〜数百枚規模になりやすく **1枚単価が効く**ため、画面でモデルを選べるように。既定の Sonnet で精度を確認し、足りなければ Opus、十分なら Haiku へ。
- **概算**（1枚 ≒ 画像1.5k + プロンプト + 出力0.5k トークン規模）: Sonnet で 1〜2円/枚 程度。バッチAPIで更に50%減。
- 思考は `thinking: {type: "adaptive"}` を既定に。科目推定など判断が要る所で有効。

### 抽出する項目（JSONの形で固定 / Pydantic）
インボイス制度・軽減税率・経費精算を踏まえた項目：

```python
class TaxLine(BaseModel):
    rate: int            # 税率(10 / 8)
    taxable_amount: int  # 対象金額(税抜)
    tax_amount: int      # 消費税額

class Receipt(BaseModel):
    used_date: str           # 利用日 YYYY-MM-DD
    vendor_name: str         # 支払先(店名)
    registration_number: str | None  # 登録番号 T+13桁（なければ null）
    total_amount: int        # 税込合計
    tax_lines: list[TaxLine] # 税率別内訳(8%混在に対応)
    payment_method: str | None   # 現金 / クレジット 等
    description: str         # 主な品目 / 摘要
    suggested_account: str   # 推定勘定科目（旅費交通費 等）
    account_reason: str      # その科目にした根拠（人の確認用）
    confidence_notes: str    # 読み取りで不確かだった点
```

- `suggested_account` は **科目候補リスト**（旅費交通費・会議費・消耗品費・接待交際費・通信費・新聞図書費…）から選ばせる。事業に合わせて編集できるようにする。
- `client.messages.parse(..., output_format=Receipt)` で結果を受け取り、金額検算（`Σtaxable + Σtax == total`）と `confidence_notes` で要確認フラグを立てる。

### プロンプト方針
- 「日本の経費精算アシスタント」と役割付け。科目候補リストを渡し「この中から最も適切なものを選び、根拠を account_reason に書く」と指示。
- 「読み取れない箇所は推測せず null / 空。曖昧な点は confidence_notes に記載」。
- 金額はカンマ・全角・通貨記号を除いた整数に正規化。軽減税率(8%)が含まれる場合は税率別に分ける。

## 6. 出力CSV

2種類を用意（画面で選択、02と同じく `st.download_button`）：

1. **経費精算書（社内提出用）**: 立替者ごと・期間ごとに 日付/支払先/科目/税込金額/摘要 を一覧化。合計付き。
2. **会計ソフト取込CSV**: freee / マネーフォワード クラウド / 弥生 形式。

- 文字コードは会計ソフト互換のため **UTF-8(BOM) / CP932(Shift_JIS)** を選択可（02も `utf-8-sig` でExcel対応）。

## 8. デプロイ / コスト / プライバシー

### 動かし方
- **ローカルで `streamlit run streamlit_app.py` して使う**のが前提。02 と同じ作り。
- APIキーは `.env`（ローカル）に置き、コードには書かない。`.gitignore` 済みにする。

### コスト
- 処理後に `usage`（入力/出力トークン）から概算費用を表示。
- 同一プロンプトの Prompt Caching、まとめ処理は Batches API（50%減）。

### プライバシー
- 経費データを外部API（Claude）へ送る点を明示。Anthropic の API はデフォルトで学習に使われない旨を README に記載。
- Webアプリなので、画像が一時的にサーバーを経由する。保存しない/処理後に消す方針を明示。

## 10. 開発ステップ

1. **足場**: 02 を参考に `streamlit_app.py` + `requirements.txt` + `.env.example` + `.streamlit/`。
2. **読み取りコア**: `src/extractor.py` を先に作り、サンプルレシートで「項目抽出 + 科目推定」が安定するまでプロンプト/スキーマを調整（Streamlit抜きで確認）。
3. **科目候補**: `accounts.py` と候補リストを整備。
4. **CSV出力**: まず会計ソフト1形式 + 経費精算書 → 検算ロジック。
5. **画面**: アップロード → 進捗付き読み取り → 編集テーブル(科目ドロップダウン) → ダウンロード を接続。
6. **要確認UX**: 行の色付け・根拠表示・手修正。
7. **デプロイ**: クラウドへ（ホスティング先確定後）。APIキーはシークレットに。
8. **拡張**: 科目の学習ルール・立替者/案件別集計・バッチ処理。

## 11. リスクと対策

| リスク | 対策 |
|---|---|
| 抽出/科目推定が100%でない | 人手修正前提のUX、科目ドロップダウン、検算フラグ、account_reason 表示 |
| レシートの多様性・感熱紙の劣化 | AIで吸収。読めない物は要確認フラグで人へ |
| 科目の判断は事業ごとに違う | 科目候補を編集可能に + 学習ルールで顧客ごとに最適化 |
| APIコスト | モデル選択、画像縮小、Caching、Batches |
| 機密データの外部送信 | キーはシークレット管理、学習不使用の明示、画像は保存しない方針 |
| 軽減税率の取り違え | 税率別内訳を必須化し、合計と検算 |

## まとめ

`02_scraping` と同じ **Streamlit のWebアプリ**として、**Claude API** でレシート/領収書を読み取り、**勘定科目まで自動仕分け**して経費精算CSV・会計取込CSVを出力する。AIはClaude一本でシンプルに作る（既定モデルは Sonnet）。MVPは「アップロード → 読み取り → 科目推定 → 編集 → CSVダウンロード」。差別化の核は **勘定科目の自動仕分け・軽減税率対応・ブラウザですぐ使える** こと。ローカルで `streamlit run` して使う（公開はしない）。
```
